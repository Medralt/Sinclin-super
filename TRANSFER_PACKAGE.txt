1. VISÃO GERAL DO SISTEMA
--------------------------------------------------------------------------------

SINCLIN é uma plataforma de saúde integrativa que opera como infraestrutura
cognitiva organizacional — não apenas um ERP ou chatbot clínico.

Três camadas:
  [UI]  sinclin-fente    — Frontend React/Vite (Lovable)
  [IA]  api.sinclin.net  — Backend Express/Node.js com GPT-4o-mini (Render)
  [BD]  Supabase         — Banco de dados PostgreSQL gerenciado em nuvem

As três instâncias são independentes e se comunicam via HTTP/REST e SDK.
Não há monorepo único: frontend e backend são repositórios git separados.

6. BACKEND — ENDPOINTS DA API (staging/api/server.js)
--------------------------------------------------------------------------------

  MÉTODO  ROTA                    DESCRIÇÃO
  ─────────────────────────────────────────────────────────────────────
  GET     /health                 Status de todos os módulos
  GET     /scanner                Introspection do runtime (uptime, mem)
  GET     /runtime                Status dos módulos (chat/sioc/session/devices)
  GET     /orchestration          Status da orquestração e agentes
  POST    /chat                   Chat IA multi-persona (GPT-4o-mini)
  POST    /sioc                   Anamnese guiada (state machine)
  GET     /sioc/:session_id       Leitura de sessão SIOC
  POST    /sioc/device/:type      Ingestão de dados de dispositivo médico
  ─────────────────────────────────────────────────────────────────────

  CONTRATO DO /chat:
    Request body aceita:
      { raw_text }          → clientes locais e SIOC
      { text, persona, session } → Lovable UI
    Persona: clinical (default) | scanner | admin | financial | marketing | orchestrator
    Response:
      { ok: true, text: "...", engine: "gpt-4o-mini", persona: "...", timestamp: "..." }
    History opcional:
      { history: [{ role: "user"|"assistant", text: "..." }] }

  SYSTEM PROMPTS POR PERSONA (PERSONAS map em server.js):
    clinical    → IA empática de saúde integrativa (default)
    scanner     → Modo analítico-técnico, classifica por severidade
    admin       → Modo administrativo, segurança e rastreabilidade
    financial   → Modo financeiro, precisão numérica e anomalias
    marketing   → Modo marketing, conversão e crescimento
    orchestrator → Visão sistêmica organizacional

7. MÓDULO SIOC — ANAMNESE GUIADA
--------------------------------------------------------------------------------

  SIOC = Sistema Integrado de Orientação Clínica

  decision.engine.js — STATELESS:
    Recebe: { session_id, input, current_step, current_data }
    Retorna: { text, next_step, structured }
    Sem estado interno — todo estado vive no session.manager

  session.manager.js — PERSISTENTE:
    L1 cache: Map em memória (TTL: 4h)
    L2 storage: Supabase (tabela sioc_sessions)
    Comportamento: upsert assíncrono não-bloqueante + re-hidratação no get
    Degradação: se SUPABASE_URL/SERVICE_KEY ausentes → memória apenas

  Máquina de estados:
    start → coletar_id → coletar_nome → coletar_idade
          → coletar_queixa → coletar_intensidade → finalizado

9. SCANNER DA UI — COMO FUNCIONA (scanner.ts)
--------------------------------------------------------------------------------

  SCANNER 1 — Runtime DOM:   detecta links para rotas não registradas
  SCANNER 2 — Rotas:         testa navegação real, classifica erros
  SCANNER 3 — Interação:     detecta botões sem handler, inputs sem label
  SCANNER 4 — Carregamento:  detecta renders vazios e timeouts
  SCANNER 5 — Dados:         duplicados, órfãos, valores negativos, vencidos

  Findings persistidos no Supabase (scanner_findings) com deduplicação.
  IA usa persona "scanner" → respostas técnicas e objetivas.
